Bridging Go to WinUI 3 / Fluent Design

Prism is written in Go. WinUI 3 is .NET/C++/COM. Here are the realistic paths:

A

Wails v3 + Fluent Web UI (Recommended)

Keep Go backend. Add Wails v3, which embeds WebView2 (Edge Chromium). Build the UI using Fluent UI web components — same look as WinUI 3, rendered in WebView2. Single binary, tray + window, acrylic via CSS. Closest to WinUI 3 possible in Go.

B

DWM Acrylic Window + Custom UI (Pure Go)

Stay in pure Go. Use DWM (Desktop Window Manager) APIs via syscall to create an acrylic-blurred window. Build settings UI with Walk or custom GDI+ rendering. Native Win32 feel, but a lot of manual UI work — controls must be hand-drawn.

C

Go + Tauri (Rust bridge)

Similar to Wails — WebView2 frontend, but Tauri's backend is Rust. Go proxy would run as a sidecar, or rewrite backend in Rust. Adds Rust toolchain dependency. WebView2 frontend gives the same Fluent Design look as Option A.

Why Option A (Wails) is recommended

Pros

  • Single Go binary — no extra runtimes
  • Full Fluent Design via HTML/CSS (acrylic, shadows, typography)
  • WebView2 is built into Windows 10/11 — no install needed
  • System tray + desktop window from one codebase
  • React/Vue/Svelte for the UI — huge ecosystem
  • ~12MB overhead on binary size

Cons

  • WebView2 dependency (preinstalled on Win 10+, but might need Edge update channel)
  • Web-based UI is not "truly native" — but visually indistinguishable with Fluent components
  • Larger refactor — current tray.go UI gets replaced entirely

What would the UI include?

Replacing the current systray-only UX with:

Settings Window

Provider selection, API keys, port config, model remapping — all in a proper window with tabs, not VBScript popups

Dashboard / Status

Connection status, uptime, request count, active models — live at a glance

System Tray

Kept minimal: quick start/stop, open dashboard, quit
